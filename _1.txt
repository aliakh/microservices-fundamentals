Below is a working, end‑to‑end setup for adding distributed logging (Loki+Promtail), monitoring (Prometheus+Grafana), and tracing (Tempo) to a Spring Boot 3.5.9 service that you run with Podman Compose. I’ve included:

podman-compose.yml (Compose v3)
Config files for Loki, Promtail, Prometheus, Tempo
Grafana provisioning for data sources (Prometheus, Loki with derived field → Tempo, Tempo)
Spring Boot Maven pom.xml deps and application.properties (Micrometer, Prometheus, OTLP/Tempo, log correlation)


Why these choices
Spring Boot 3+ uses Micrometer for metrics and tracing; you expose /actuator/prometheus for Prometheus and export traces via OTLP directly to Tempo (gRPC 4317). That’s the documented path in Spring Boot and Tempo.
For logs, we use Promtail to tail Podman’s container log files and push them into Loki; Grafana then links logs→traces using a derived field that extracts the traceId from your app log line.
Heads‑up: Promtail is in LTS until Mar 2, 2026; Grafana recommends migration planning (e.g., to Grafana Alloy) for the future. This stack still works fine today. [docs.spring.io], [docs.spring.io], [grafana.com] [grafana.com], [grafana.com] [grafana.com]


1) Podman Compose file
Save as podman-compose.yml (Compose v3 syntax works with Podman’s compose wrapper). If you normally run Docker Compose files already, Podman supports that workflow with its Docker‑compatible socket/CLI. [docs.podman.io], [redhat.com], [podman-desktop.io]

Notes

Podman Compose is a thin wrapper; using Compose v3 YAML like above is fine. [docs.podman.io]
We mount both rootful and rootless Podman container log directories so Promtail can tail ctr.log files. Podman stores container logs as ctr.log under overlay-containers/<id>/userdata/ (paths differ rootful vs. rootless). [access.redhat.com], [stackoverflow.com], [support.scc.suse.com]

3) Promtail config (promtail/promtail-config.yaml)
Tails system logs and Podman’s ctr.log files and pushes to Loki.
(If you only run rootless/rootful, you can remove the other glob.)
YAMLserver:  http_listen_port: 9080  grpc_listen_port: 0positions:  filename: /tmp/positions.yamlclients:  - url: http://loki:3100/loki/api/v1/pushscrape_configs:  # Tail system logs as an example  - job_name: varlogs    static_configs:      - targets: [localhost]        labels:          job: varlogs          __path__: /var/log/*.log  # Tail Podman ROOTFUL container logs (ctr.log)  - job_name: podman-rootful    static_configs:      - targets: [localhost]        labels:          job: podman          # ctr.log files live under overlay-containers/*/userdata/          __path__: /var/lib/containers/storage/overlay-containers/*/userdata/ctr.log  # Tail Podman ROOTLESS container logs (ctr.log)  - job_name: podman-rootless    static_configs:      - targets: [localhost]        labels:          job: podman          __path__: ${HOME}/.local/share/containers/storage/overlay-containers/*/userdata/ctr.logShow less

Why these paths? Podman writes stdout/stderr to ctr.log; the exact path varies. Red Hat’s guidance and distro KBs document overlay-containers/<id>/userdata/ctr.log for rootful/rootless.

4) Tempo config (tempo/tempo.yaml)
Enable OTLP gRPC (4317) and HTTP (4318) receivers; local filesystem storage; single‑binary mode. Grafana’s Tempo docs show gRPC 4317 as the modern default. [grafana.com], [github.com]
YAMLserver:  http_listen_port: 3200distributor:  receivers:    otlp:      protocols:        grpc:          endpoint: 0.0.0.0:4317        http:          endpoint: 0.0.0.0:4318    zipkin:    jaeger:      protocols:        thrift_http:querier:  frontend_worker:    frontend_address: 127.0.0.1:9095query_frontend:  search:    max_duration: 1hstorage:  trace:    backend: local    local:      path: /tmp/tempo    wal:      path: /tmp/tempo/walcompactor:  compaction:    block_retention: 24hShow less

Tempo’s quickstart and configuration show enabling both otlp.grpc and otlp.http receivers, with Grafana connecting via Tempo’s HTTP API on 3200. [grafana.com], [grafana.com]

6) Grafana provisioning (data sources)
Put these under grafana/provisioning/datasources/datasources.yml. Grafana provisioning system is YAML-based; this creates Prometheus (default), Loki (with derived field that links logs to Tempo traces), and Tempo data sources. [grafana.com], [grafana.com], [grafana.com]
YAMLapiVersion: 1datasources:  - name: Prometheus    type: prometheus    access: proxy    url: http://prometheus:9090    isDefault: true    jsonData:      httpMethod: POST  - name: Loki    type: loki    access: proxy    url: http://loki:3100    jsonData:      maxLines: 1000      # Link logs to traces via traceId extracted from log line      derivedFields:        - name: TraceID          matcherRegex: "traceId=(\\w+)"   # matches our Spring log pattern below          datasourceUid: tempo_ds          # must match Tempo's uid          url: "$${__value.raw}"           # pass traceId to Tempo datasource  - name: Tempo    uid: tempo_ds    type: tempo    access: proxy    url: http://tempo:3200    jsonData:      tracesToLogsV2:        # optional: let Tempo link back to logs in Loki using the same traceId label        datasourceUid: loki        filterByTraceID: true        spanStartTimeShift: "-1h"        spanEndTimeShift: "1h"Show less

References: Grafana provisioning docs and Loki derivedFields to build the log→trace link; the Tempo data source docs for querying traces. [grafana.com], [grafana.com], [grafana.com]


7) Spring Boot: Maven pom.xml
These dependencies enable Actuator metrics, Prometheus registry, Micrometer Tracing with OpenTelemetry bridge, and the OTLP exporter that sends traces to Tempo. This matches Spring Boot’s tracing guidance (Micrometer + OTLP). Do not set versions; let Spring Boot’s BOM manage them. [docs.spring.io], [baeldung.com]
XML<dependencies>  <!-- Actuator endpoints, including /actuator/prometheus -->  <dependency>    <groupId>org.springframework.boot</groupId>    <artifactId>spring-boot-starter-actuator</artifactId>  </dependency>  <!-- Micrometer Prometheus registry -->  <dependency>    <groupId>io.micrometer</groupId>    <artifactId>micrometer-registry-prometheus</artifactId>  </dependency>  <!-- Micrometer Tracing with OpenTelemetry bridge -->  <dependency>    <groupId>io.micrometer</groupId>    <artifactId>micrometer-tracing-bridge-otel</artifactId>  </dependency>  <!-- OpenTelemetry OTLP exporter (traces) -->  <dependency>    <groupId>io.opentelemetry</groupId>    <artifactId>opentelemetry-exporter-otlp</artifactId>  </dependency>  <!-- Your web starter, etc. -->  <dependency>    <groupId>org.springframework.boot</groupId>    <artifactId>spring-boot-starter-web</artifactId>  </dependency>  <!-- (Optional) JDBC, JPA, etc. --></dependencies>Show less

Spring Boot 3.x exposes Prometheus metrics when you include Actuator + Micrometer Prometheus; Micrometer Tracing + OTLP exporter is the standard for OpenTelemetry/Tempo. [docs.spring.io], [docs.micrometer.io], [baeldung.com]


8) Spring Boot: src/main/resources/application.properties

Expose /actuator/prometheus
Enable tracing, set sampling rate for validation
Send spans via OTLP/gRPC to tempo:4317
Add log correlation pattern (traceId, spanId) so Grafana/Loki can create a clickable link to Tempo using the derived field.

Plain Textproperties isn’t fully supported. /actuator/prometheus is the documented scrape endpoint. management.otlp.tracing.endpoint sends Micrometer spans to OTLP; Tempo’s gRPC and HTTP receivers live on 4317/4318. [docs.spring.io], [baeldung.com], [grafana.com]

10) Troubleshooting & tips

No traces in Tempo? Ensure your exporter points to gRPC 4317 and that Tempo has otlp.grpc enabled; OTLP supports gRPC (4317) and HTTP (4318) — they must match. [grafana.com], [oneuptime.com]
Prometheus scrape fails? Verify /actuator/prometheus is exposed and the Prometheus registry dependency is on the classpath. [docs.spring.io], [docs.micrometer.io]
Promtail sees no Podman logs? Confirm the ctr.log paths exist for your Podman mode (rootful vs. rootless). You can discover a specific container’s log file via:
podman inspect --format='{{.HostConfig.LogConfig.Path}}' <container> [access.redhat.com]
Log file growth: set Podman’s log_size_max in containers.conf to rotate/limit ctr.log. [support.scc.suse.com]
Compose & Podman: If you prefer docker compose syntax, Podman supports a Docker‑compatible socket so the same YAML works. [podman-desktop.io]
Future‑proofing: Promtail is LTS/EOL in 2026 — consider migrating to Grafana Alloy for log/trace/metric pipelines later. [grafana.com]
