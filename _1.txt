

5) Prometheus config (prometheus/prometheus.yml)
Scrape the Spring Boot app’s /actuator/prometheus endpoint and (optionally) Loki/Tempo/Promtail internal metrics. Spring Boot’s actuator endpoint and Micrometer Prometheus registry are the standard way to expose metrics. [docs.spring.io], [docs.micrometer.io]
global:
  scrape_interval: 15s
  evaluation_interval: 15s

scrape_configs:
  - job_name: spring-app
    static_configs:
      - targets: ["app:8080"]
    metrics_path: /actuator/prometheus

  - job_name: loki
    static_configs:
      - targets: ["loki:3100"]

  - job_name: tempo
    static_configs:
      - targets: ["tempo:3200"]

  - job_name: promtail
    static_configs:
      - targets: ["promtail:9080"]

6) Grafana provisioning (data sources)
Put these under grafana/provisioning/datasources/datasources.yml. Grafana provisioning system is YAML-based; this creates Prometheus (default), Loki (with derived field that links logs to Tempo traces), and Tempo data sources. [grafana.com], [grafana.com], [grafana.com]
apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    jsonData:
      httpMethod: POST

  - name: Loki
    type: loki
    access: proxy
    url: http://loki:3100
    jsonData:
      maxLines: 1000
      # Link logs to traces via traceId extracted from log line
      derivedFields:
        - name: TraceID
          matcherRegex: "traceId=(\\w+)"   # matches our Spring log pattern below
          datasourceUid: tempo_ds          # must match Tempo's uid
          url: "$${__value.raw}"           # pass traceId to Tempo datasource

  - name: Tempo
    uid: tempo_ds
    type: tempo
    access: proxy
    url: http://tempo:3200
    jsonData:
      tracesToLogsV2:
        # optional: let Tempo link back to logs in Loki using the same traceId label
        datasourceUid: loki
        filterByTraceID: true
        spanStartTimeShift: "-1h"
        spanEndTimeShift: "1h"

References: Grafana provisioning docs and Loki derivedFields to build the log→trace link; the Tempo data source docs for querying traces. [grafana.com], [grafana.com], [grafana.com]


7) Spring Boot: Maven pom.xml
These dependencies enable Actuator metrics, Prometheus registry, Micrometer Tracing with OpenTelemetry bridge, and the OTLP exporter that sends traces to Tempo. This matches Spring Boot’s tracing guidance (Micrometer + OTLP). Do not set versions; let Spring Boot’s BOM manage them. [docs.spring.io], [baeldung.com]
<dependencies>
  <!-- Actuator endpoints, including /actuator/prometheus -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-actuator</artifactId>
  </dependency>

  <!-- Micrometer Prometheus registry -->
  <dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-registry-prometheus</artifactId>
  </dependency>

  <!-- Micrometer Tracing with OpenTelemetry bridge -->
  <dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-otel</artifactId>
  </dependency>

  <!-- OpenTelemetry OTLP exporter (traces) -->
  <dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-exporter-otlp</artifactId>
  </dependency>

  <!-- Your web starter, etc. -->
  <dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
  </dependency>

  <!-- (Optional) JDBC, JPA, etc. -->
</dependencies>

Spring Boot 3.x exposes Prometheus metrics when you include Actuator + Micrometer Prometheus; Micrometer Tracing + OTLP exporter is the standard for OpenTelemetry/Tempo. [docs.spring.io], [docs.micrometer.io], [baeldung.com]


8) Spring Boot: src/main/resources/application.properties

Expose /actuator/prometheus
Enable tracing, set sampling rate for validation
Send spans via OTLP/gRPC to tempo:4317
Add log correlation pattern (traceId, spanId) so Grafana/Loki can create a clickable link to Tempo using the derived field.

# --- Basics ---
spring.application.name=demo-service

# --- Actuator / Prometheus ---
management.endpoints.web.exposure.include=health,info,metrics,prometheus
management.endpoint.health.show-details=always

# --- Micrometer Tracing ---
management.tracing.enabled=true
# For local validation; in prod use 0.1-0.3 typically
management.tracing.sampling.probability=1.0

# --- OpenTelemetry / OTLP exporter (Tempo @ gRPC 4317) ---
management.otlp.tracing.endpoint=http://tempo:4317

# --- Log correlation (SLF4J MDC adds traceId/spanId to logs) ---
# This ensures lines like: INFO [demo-service,abc123...,def456...] ...
logging.pattern.level=%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]

Plain Textproperties isn’t fully supported. 10) Troubleshooting & tips

No traces in Tempo? Ensure your exporter points to gRPC 4317 and that Tempo has otlp.grpc enabled; OTLP supports gRPC (4317) and HTTP (4318) — they must match. [grafana.com], [oneuptime.com]
Prometheus scrape fails? Verify /actuator/prometheus is exposed and the Prometheus registry dependency is on the classpath. [docs.spring.io], [docs.micrometer.io]
Promtail sees no Podman logs? Confirm the ctr.log paths exist for your Podman mode (rootful vs. rootless). You can discover a specific container’s log file via:
podman inspect --format='{{.HostConfig.LogConfig.Path}}' <container> [access.redhat.com]
Log file growth: set Podman’s log_size_max in containers.conf to rotate/limit ctr.log. [support.scc.suse.com]
Compose & Podman: If you prefer docker compose syntax, Podman supports a Docker‑compatible socket so the same YAML works. [podman-desktop.io]
Future‑proofing: Promtail is LTS/EOL in 2026 — consider migrating to Grafana Alloy for log/trace/metric pipelines later. [grafana.com]
